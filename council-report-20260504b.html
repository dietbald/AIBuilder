AIBuilder Council Report — Redesigned Architecture

Second council session — evaluating the cron + persistent interactive session design
Question: The previous design's fatal bugs are fixed. The new design uses persistent interactive claude sessions triggered by cron via tmux send-keys. Each sub-agent gets its own tmux session. Is this architecture sound, and what breaks on first run?
5 advisors · 5 peer reviewers · 1 chairman synthesis · 2026-05-04
The One Thing to Do First

Verify the foundational assumption before writing another line of code

Run this test. If the output contains "PONG" cleanly, the tmux send-keys trigger is viable. If not, the mechanism must be replaced before anything else.

tmux new-session -d -s test-claude
tmux send-keys -t test-claude "claude" Enter
sleep 5
tmux send-keys -t test-claude "say the word PONG and nothing else" Enter
sleep 15
tmux capture-pane -t test-claude -p

Council Verdict

Where the Council Agrees

4 of 5 advisors (Contrarian, First Principles, Outsider, Executor — not the Expansionist) independently flagged two structural issues:

1. tmux send-keys is not a message bus. It sends keystrokes to a terminal pane with no acknowledgment, no error, and always exits 0. The cron job cannot distinguish "delivered and processed" from "injected into live output," "sent to dead session," or "queued behind two prior unprocessed ticks." Silent failure is the default, not the exception. (The Expansionist called this "an opportunity" — the council dismissed that framing.)

2. Persistent conversation history is a liability with an expiration date. After enough ticks, context degrades. The conductor weights hallucinated history over STATUS.md ground truth. This doesn't announce itself — it looks like drift, then flakiness, then confident wrong decisions. No clean recovery without destroying the context. (The Expansionist called this "emergent intelligence" — the council disagreed: it is a shrinking context window, not accumulating intelligence.)

2 of 5 advisors (Outsider, Executor) explicitly flagged a first-run operational break:

3. Cron PATH will break on first run. Fix: add PATH=/usr/bin:/bin at the top of the crontab before anything else runs.

Where the Council Clashes

Is persistent conversation history an upgrade or a liability? The Expansionist argued it enables "emergent intelligence" — the conductor builds a mental model, gets smarter over time. The remaining four advisors disagreed. LLM conversation history is not cumulative learning; it is a growing context window with a fixed ceiling. The conductor does not get smarter; it gets less reliable. The Expansionist's own proposed solution (session rotation) destroys the accumulated context it claimed was the advantage.

How severe is tick-doubling? The Executor called it an "acceptable risk." Peer reviewers pushed back: on a 3-minute cron interval, a slow LLM response will eventually cause tick stacking — conductor processes tick N+1 without completing tick N's dispatches. On a long-running project this is a guaranteed event, not an edge case.

Blind Spots the Council Caught

Highest-confidence finding (caught by all 5 reviewers independently): Does claude interactive mode behave predictably when stdin is a tmux pane rather than a real terminal? PTY vs pipe semantics, input buffering, output flushing, signal handling — none of this is specified in the Claude CLI. The entire architecture rests on an undocumented assumption. If claude buffers input differently under pseudo-TTY conditions, everything collapses before any other failure mode matters.

⭐ STRESS TEST #1 — This is the first test to execute on the platform. The PONG test (see "The One Thing to Do First" above) must pass before any further development. Result determines whether tmux send-keys remains the trigger mechanism or must be replaced with a file-based queue.

API errors kill the process, not the session. A rate limit, 5xx, or mid-tick timeout leaves the tmux session alive but the Claude process exited or hung. Cron fires into it. No watchdog, no restart policy, no way to distinguish "thinking" from "dead."

No recovery procedure for mid-task conductor death. Sub-agent completes and writes output. Conductor is gone. STATUS.md goes stale. The next tick fires into a fresh session with no context of orphaned agents. No re-entry checkpoint reconciles in-flight sessions with STATUS.md state.

Tick stacking has no debounce. If two cron firings queue before the first is consumed, conductor processes tick 2 without completing tick 1's dispatches. No deduplication mechanism exists.

The Recommendation

The architecture is a meaningful improvement. The fatal bugs from the first council are fixed: the loop exists, session names are correct, no more single window per session, double-dispatch is addressed. The structural skeleton is sound.

It is not production-ready. Fix in this order:

  1. Run the PONG test. Verify claude interactive works correctly under tmux pseudo-TTY injection. If it fails, replace the trigger mechanism with a file-based queue or named pipe before proceeding.
  2. Add cron PATH. PATH=/usr/bin:/bin in crontab. Do this before first run.
  3. Fix tmux exact-match. Use tmux has-session -t "=session-name" (note the = prefix, tmux 3.x+) for the double-dispatch check. Without it, partial name matches cause false negatives.
  4. Replace session-existence check with status files. tmux has-session tells you a session exists, not that the agent is alive or finished. Write .devloop/agent-status/${ROLE}-${FEATURE}.status with states: running, complete, failed. Dispatch checks the file, not the session.
  5. Add tick deduplication. Write a lockfile at tick start, remove at tick end. If cron fires and lockfile exists, skip. Prevents tick stacking without IPC changes.
  6. Add conductor watchdog to co-conductor. Verify the conductor session is responsive, not just present. A dead session that the co-conductor "audits" produces logs, not recovery.
  7. Add a restart reconciliation procedure. When the conductor session restarts (by co-conductor or manually), it must scan .devloop/agent-status/ for orphaned running agents, check their output files, and advance STATUS.md before starting the next tick.

Full Advisor Responses
Outsider The one-shot approach failed loudly. This one fails silently.

tmux send-keys "tick" Enter is not a reliable IPC mechanism. It sends keystrokes to whatever is currently focused in that pane. If Claude is mid-response when cron fires, you've injected "tick" into the middle of output. If the session crashed and restarted, you're sending to a fresh context with no memory. If the pane is at a prompt but Claude hung waiting on a subprocess, "tick" queues silently and fires at the wrong moment. You have no acknowledgment, no error, no log entry. The cron job always exits 0.

Conversation history is a liability disguised as a feature. The conductor "remembering what it dispatched" sounds good until the context window fills up, the model hallucinates a state that contradicts STATUS.md, or a corrupted turn poisons everything downstream. Two sources of truth — when they conflict, the model decides, inconsistently.

The double-dispatch check is correct but incomplete. tmux has-session tells you the session exists, not that the agent is still running. A finished agent whose session wasn't cleaned up blocks redispatch forever.

First-run break: cron PATH won't have tmux. Add PATH=/usr/bin:/bin explicitly in crontab.

The co-conductor's STATUS.md snapshot before audit — who writes it? If the conductor is mid-tick when audit fires, you get a partial snapshot. Race condition, no lock.

Expansionist Persistent session is an underrated upgrade — stateful reasoning accumulation.

A conductor with persistent conversation history doesn't just remember what it dispatched — it builds a mental model of the project over time. It can notice "three failed QA runs on authentication in the last six ticks" without that insight being written anywhere in STATUS.md. This is emergent intelligence one-shot designs permanently forfeit.

The tmux send-keys trigger is an opportunity. Because the conductor is persistent, you could send it anything: a Slack message, GitHub webhook payload, a user override command. The cron job is just the heartbeat. The conductor is now always listening.

Dedicated session per sub-agent scales toward parallelism — nothing prevents dispatching a spec-author and a reviewer simultaneously on different features. Parallel feature development is already latent in this design.

The context window length limit is the one real constraint — solvable with periodic session rotation, and a good problem to have.

Contrarian Worst of both worlds — fragility of stateful, none of the durability.

Conversation history across ticks sounds like memory, but it's actually drift. After 50+ ticks, the conductor reasons against a polluted context that contradicts the current STATUS.md. You can't flush it without killing the session — and killing destroys the memory. The architecture inherits the worst of both worlds.

tmux send-keys fires and forgets. There is no acknowledgment. If Claude is processing a previous tick, the new "tick" gets injected into live output. You now have two concurrent tick-processing threads inside one conversation. The double-dispatch prevention on sub-agents doesn't help — the conductor itself is the race condition.

The session-existence check is too coarse. A crashed session may linger. A completed --print run may or may not clean up its tmux session. You need explicit status files, not session polls.

No heartbeat, no watchdog. If the conductor dies silently — OOM, API timeout, network drop — the cron job keeps firing into a dead session. The co-conductor auditing a broken conductor will just document the silence.

First Principles Wrong control flow — terminal is a human interface, not a message bus.

The real problem is maintaining durable, resumable state across an AI agent that has no native persistence. Every design decision here is a workaround for that fundamental gap.

tmux send-keys delivers keystrokes to a terminal, not stdin to a process. If the Claude session is mid-output when cron fires, the "tick" string gets injected into the output stream. This is not a race condition you can tune away — it's structural.

The deeper wrong question: why is a cron job reaching into a terminal to poke an AI? The conductor should pull triggers from a queue it owns — a file, a socket, a named pipe — not receive injected keystrokes from outside.

Persistent interactive sessions accumulate context. STATUS.md as "authoritative source" partially mitigates this, but the conductor will increasingly weight hallucinated history over the file as context grows.

Real question: should the conductor be interactive at all? Consider externalizing all state to files and using claude --resume or fresh invocations per tick instead.

Executor Mostly sound — three concrete first-run breaks.

Break #1: Session target ambiguity. tmux has-session -t agent-spec-author-auth-BidPlatform matches any session whose name starts with that string. Use the = prefix: tmux has-session -t "=agent-spec-author-auth-BidPlatform" (tmux 3.x+). Without this, double-dispatch prevention is unreliable.

Break #2: Interactive session context drift. After 20-30 ticks the conductor's conversation fills with STATUS.md dumps and dispatch history. Token pressure degrades reasoning quality and eventually hits context limits. Fix: add a tick counter to STATUS.md; at tick 50, restart the conductor session clean.

Break #3: Cron PATH. Add PATH=/usr/bin:/bin in crontab.

If cron fires while conductor is mid-response, tmux send-keys queues the tick in the terminal buffer. Claude sees it after finishing. Usually fine — can cause tick-doubling under slow dispatch. Acceptable risk for now.

Everything else — dedicated sessions per sub-agent, STATUS.md as truth source, has-session gating — is structurally correct.

Peer Review Highlights (5 reviewers)

Strongest (unanimous 5/5): Outsider (Response A) — identified the most consequential failure modes with precision. Every point is falsifiable. The "fails silently" framing is the key insight.

Biggest blind spot (unanimous 5/5): Expansionist (Response B) — rebrands every structural flaw as an opportunity. "Emergent intelligence" is a false claim — LLM conversation history is a growing context window that degrades, not cumulative learning. "Session rotation is solvable" destroys its own thesis by flushing the accumulated context.

What all five advisors missed (caught by peer review):

  • Claude CLI PTY behavior under tmux injection is undocumented (all 5 reviewers): does interactive claude buffer input, flush output, handle signals differently under pseudo-TTY conditions? This is the foundational unknown the entire architecture rests on.
  • API errors kill the process, not the session (3 reviewers): rate limit or 5xx leaves the tmux session alive but Claude exited. Cron fires into it. No distinction between "thinking" and "dead."
  • Tick stacking and no debounce (2 reviewers): two queued ticks before first is consumed → conductor processes tick 2 without resolving tick 1. Guaranteed on any slow response on a 3-minute interval.
  • No re-entry checkpoint after conductor restart (2 reviewers): sub-agent completes mid-crash, output written, nobody reads it. STATUS.md stale. No reconciliation procedure.
