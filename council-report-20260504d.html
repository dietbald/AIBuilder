✅ Where the Council Agrees

The system fails silently. Every advisor independently arrived at this conclusion from a different angle. The Contrarian catalogued seven specific silent failures. The Outsider named it the core finding. The Executor wrote fixes for it. The First Principles Thinker called it "confident wrongness at scale." No advisor found a counter-example. When AIBuilder breaks, nothing tells you — not the user, not the logs, not the STATUS.md, not the agents. Silence is indistinguishable from success.

The Contrarian's seven bugs are confirmed. All five peer reviewers ranked the Contrarian's response as strongest. The bugs were not challenged on accuracy — only criticized for insufficient scope. They should be treated as confirmed failure modes, not allegations.

The Codex documentation is actively harmful. Stale Codex references don't just mislead developers — they break the Reviewer's stated preconditions, making its behavior undefined in the actual operating environment (Claude Sonnet, not Codex).

⚡ Where the Council Clashes

Is this a pipeline problem or a search problem?
The First Principles Thinker argues the linear pipeline framing is the root cause — software development is recursive and uncertain; forcing it into sequential stages produces the wrong architecture. The Executor implicitly rejects this: the implementation has fixable bugs, not a conceptual flaw. These are irreconcilable without defining what "success" means. For known-domain features with clear specs, the pipeline works. For novel or ambiguous requirements, the pipeline will produce confident wrong output.

Is the Expansionist's vision useful?
All five reviewers called Response C the biggest blind spot — vision masquerading as technical analysis. The factory/portfolio framing is not wrong, but is irrelevant when the system deadlocks on an empty STATUS.md before running a single feature. Fix the car before planning the road trip.

🔍 Blind Spots the Council Caught (Peer Review Round)

These issues only emerged through the peer review phase — no advisor identified them initially:

  • No inter-agent output validation. No schema, checksum, or handshake between stages. A malformed spec propagates silently through Implementer, Reviewer, and QA Tester. File presence is treated as correctness.
  • No observability infrastructure. No structured event log, no flight recorder. When the system fails silently, there is no artifact to diagnose from. Post-mortem is impossible.
  • No idempotency or atomic writes. An interrupted agent leaves a half-written file. The next stage reads it without complaint. A crashed Implementer corrupts its artifact; the Reviewer reviews it confidently.
  • No recovery semantics. Mid-pipeline failures leave STATUS.md, DECISIONS.md, and retry counters in indeterminate state. No rollback. No checkpoint. The next run inherits garbage state from the previous failed run.
  • Write-only feedback loop. Each agent writes output but does not read predecessor reasoning. Compounding errors are invisible because no agent has visibility into why the prior stage made its choices.
📋 The Recommendation

Do not run AIBuilder on any real feature until the known bugs and the five architectural gaps are addressed.

Priority 1 — Fix the 7 known bugs (Executor's list, ordered by kill-on-run-day impact). The system cannot demonstrate any other property if it deadlocks on startup. These are the pre-requisites.

Priority 2 — Add observability before running any real workload. A structured event log is the minimum prerequisite for trusting system output. Every other fix is partially blind without it.

Priority 3 — Atomic writes and recovery semantics. Without these, unattended operation will eventually corrupt state and produce garbage output that the system cannot detect or discard.

The pipeline architecture is not wrong in principle for known-domain features. But it is missing the infrastructure that makes autonomous systems trustworthy: the ability to detect its own failures, record what happened, refuse corrupt inputs, and recover to a known-good state.

⚡ The One Thing to Do First

Add a structured append-only event log.

Every agent, Conductor, and cron job writes one line on each state transition: timestamp, agent role, feature ID, status, action. This is the single change that makes every other problem diagnosable. Without it, you cannot confirm any fix worked. You cannot detect the retry key collision. You cannot see the STATUS.md deadlock happening. You cannot distinguish silent success from silent failure.

It costs one log-write call per agent output block and one shared log path in the environment. Do this before running the system on any real feature.

Full Advisor Responses
The Contrarian 7 ranked failure modes — all confirmed by peer review

1. QA Tester dead on arrival. BrowserControl not in tools frontmatter. Agent fails every UI test silently or with bash errors. Every feature cycles through retries → Tier 4. Not a doc bug — a broken agent.

2. E2E tests run against nothing. No dev server before test-authoring. "All tests GREEN" is structurally impossible to satisfy.

3. Retry budget poisoning is silent and guaranteed. 2 implementing failures + 1 review failure = immediate Tier 4 via shared counter key. Not an edge case — the default path for non-trivial features.

4. Fresh project deadlocks silently. Conductor logs "idle" forever. No trigger to populate STATUS.md from Inception output. Pipeline never starts.

5. Codex references are live stale state. Reviewer instructed "if Implementer ran on Codex..." — permanently false precondition. Reviewer behavior is undefined in practice.

6. AskUserQuestion blocks unattended spec author. Tool doesn't exist. 30-min watchdog kills it. Wasted retry every time a spec has ambiguity.

7. Session rotation at tick 50 drops in-flight completions. Nothing drains active agents before self-termination. Completions written during restart window get dropped.

First Principles Software development is not a pipeline

The central assumption is wrong. A pipeline implies deterministic stage transitions. Software development is a search problem under uncertainty — specs reveal ambiguities when implementation begins; tests expose requirements the spec never articulated.

The system treats STATUS.md as ground truth. Ground truth is the running system. Every abstraction layer is a lossy encoding; errors compound. By QA Tester stage, the original intent has been telephone-gamed through four agents.

The most dangerous failure mode: confident wrongness at scale — coherent, test-passing output that solves the wrong problem. No stage asks "is this what we needed?"

The right question: "Where does human judgment re-enter the loop, and is that checkpoint load-bearing or ceremonial?"

Expansionist The team is building a factory, not a tool

The architecture — file-based state, restartable agents, pipeline stages — is a reusable manufacturing line. Multiple parallel Conductors = N products simultaneously. One server, infinite products.

Pipeline termination at done is the wrong endpoint. Real software evolves. Post-deploy monitoring feeding back into FEATURES.md = living system.

DECISIONS.md is undervalued institutional memory. Indexable across projects, the system gets smarter with each project — cross-project learning, the Conductor stops starting from zero.

The 10x framing isn't "build software faster." It's autonomous product portfolio management.

Note: All five peer reviewers identified this response as the biggest blind spot — vision without engagement with actual failure modes.

The Outsider Every failure is silent — user has no diagnostic signal

Following the docs exactly:

Phase 1: The BA agent calls AskUserQuestion. The tool doesn't exist. The agent fails mid-inception. The error looks like a tool resolution failure — user assumes they installed something wrong.

Phase 2: devloop-start.sh creates STATUS.md with empty headers. Conductor reads it, finds nothing, logs "idle" forever. System appears running. Nothing happens. No error. A first-time user waits indefinitely.

This is the worst kind of failure: silent, plausible-looking, and pointing nowhere.

Stale Codex docs give readers a completely wrong mental model. BrowserControl silently unavailable. Core problem: the system provides no indication of where it is or what broke.

The Executor 5 concrete fixes ordered by kill-on-run-day priority

1. STATUS.md deadlock — In devloop-start.sh, after STATUS.md init: grep -oE '^## (F-[0-9]+)' "$PROJECT_DIR/FEATURES.md" | sed 's/## //' | while read F; do echo "| $F | pending |" >> STATUS.md; done

2. AskUserQuestion — Remove from dev-spec-author.md tools list. Replace body references (lines 77, 93) with "document in [NEEDS CLARIFICATION] section."

3. No server before e2e — In dev-test-author.md: add step to read CODING_STANDARDS.md for server start command, start in background, wait for reachability, then run tests.

4. Retry counter collision — Use ${FEATURE}:${ROLE}:review-retry as key when re-dispatching from review failure; separate from implementing-stage counter.

5. Wrong metadata — Add BrowserControl to QA Tester frontmatter tools. Replace Codex refs with actual models (claude-sonnet-4-6, claude-opus-4-7). Update test-author description from "parallel" to "sequential."
